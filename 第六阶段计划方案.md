## 3. 技术方案设计

### 3.1 质量门禁策略

第六阶段采用“三层验证”：

1. **现有单元/集成测试层**
   - 继续使用 Vitest 验证搜索、推荐、API、页面组件
   - 保持当前定向测试集合作为回归基础

2. **端到端流程层**
   - 增加浏览器级 E2E 测试
   - 覆盖首页 -> 搜索 -> 详情 -> 已读/收藏 -> 用户中心主链路
   - 覆盖标签页 -> 标签详情 -> 分页/排序链路
   - 覆盖桌面启动前提下本地 API 可用性验证

3. **桌面手工验收层**
   - 启动安装包
   - 检查首页、搜索、详情、标签、用户中心
   - 验证本地 API、历史、收藏、偏好保存、桌面退出重启

### 3.2 E2E 工具路线

结合当前仓库状态和第六阶段目标，建议：

| 目标 | 选型 | 说明 |
|------|------|------|
| Web UI 端到端验证 | Playwright | 适合验证前端与本地 API 的真实页面交互 |
| 桌面验收 | 手工清单 + 启动脚本 | 首版先确保稳定验收，不在本阶段强求完整桌面自动化 |
| 接口冒烟 | 复用现有 Vitest / API 测试 | 避免把所有验证都堆到 E2E 层 |

**这样拆分的原因**：

- 当前项目已有 React + Vite 前端，Playwright 接入成本可控
- Electron 完整自动化成本较高，不适合在首版发布前强行扩展过深
- 第六阶段的核心是“首版可交付”，不是“把所有自动化一次做满”

### 3.3 桌面发布路线

第五阶段已经落地 Electron 路线，因此第六阶段不再切换技术栈，而是补齐发布闭环：

- 保持 **Electron** 作为首版桌面容器
- 引入安装包构建工具（如 `electron-builder`）
- 先打通 **Windows 首发链路**
- macOS 兼容性作为后续明确目标保留，但不作为本阶段阻断条件

**范围控制理由**：

- 当前需求已经明确指向 Windows 与 macOS，其中 Windows 为第一优先级
- 首版目标是先把一个可安装、可运行、可验证的 Windows 版本稳定交付
- macOS 支持需要后续补齐签名、打包与兼容性验证，适合放到下一阶段继续推进

### 3.4 性能与成本调优路线

第六阶段关注以下可量化指标：

| 指标 | 目标 |
|------|------|
| 本地 API 冷启动完成 | < 5s |
| 首页首屏主要接口完成 | < 1.5s |
| 搜索结果返回并渲染 | < 2s |
| 标签页切换与分页响应 | < 1s |
| 详情页打开并展示内容 | < 1s |
| 常规回归测试总时长 | 控制在 5 分钟内 |

成本与资源方面重点检查：

- 摘要是否重复生成
- 向量索引是否存在重复全量扫描
- 调研报告生成是否有明确触发阈值
- 桌面端重启后是否会重复执行高成本初始化

### 3.5 文档交付路线

第六阶段文档不追求“大而全”，而是围绕首版交付需要组织：

- 开发者启动说明
- 打包说明
- 首版发布说明
- 故障排查清单
- 手工验收清单

文档目标读者：

- 项目内部开发者
- 下一个接手发布流程的工程师

文档写作原则：

- 优先可执行步骤
- 尽量避免空泛架构描述
- 所有发布说明都要能支持“陌生读者照着操作”

## 5. 测试与验收标准

### 5.1 功能验收

| 验收项 | 目标 |
|--------|------|
| 首页 | 可正常展示热点推荐、个性化推荐和最近更新 |
| 搜索页 | 可完成搜索、筛选、分页和报告查看 |
| 详情页 | 可查看摘要并执行收藏/已读动作 |
| 标签页 | 可浏览标签、切换分类、排序和分页 |
| 用户中心 | 可查看并修改兴趣、记忆、历史、收藏、偏好 |
| 桌面启动 | 可通过桌面入口启动并连通本地 API |
| 打包产物 | 可生成可分发的 Windows 桌面安装包或等价发布产物 |

### 5.2 测试清单

- 自动化测试
  - 页面级 smoke test
  - 关键 API 回归
  - E2E 主链路测试
  - 打包前构建验证

- 手工测试
  - 桌面启动与退出
  - 首次打开首页
  - 搜索与分页
  - 详情页已读/收藏
  - 用户偏好保存与重启后保留

### 5.3 发布门禁

满足以下条件才允许进入首版发布：

- `typecheck` 通过
- 核心 Vitest 回归通过
- E2E smoke 通过
- 桌面构建产物生成成功
- 手工验收 checklist 无 blocker
- 已知问题列表中不存在 P0 缺陷

## 6. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| Electron 打包配置引入不稳定性 | 影响首版交付时间 | 先打通 Windows 单平台，避免多平台同时收口 |
| E2E 测试环境搭建过重 | 拖慢第六阶段节奏 | 先做主链路 smoke，不在本阶段追求全量覆盖 |
| 桌面 GUI 仍有人工验证缺口 | 发布时可能遗漏真实交互问题 | 保留手工验收清单作为发布门禁 |
| 性能问题集中暴露 | 影响首版体验 | 只修阻断与高频问题，把深度优化留到第七阶段 |
| 发布文档不完整 | 后续接手者难以复现 | 文档与打包脚本同步交付，避免“口头流程” |
