# PaperHub 第五阶段计划方案：前端界面与桌面端集成

> **阶段定位**：承接第四阶段（推荐系统与用户记忆），聚焦 **首页**、**搜索页**、**文章详情页**、**标签页**、**用户中心** 五个核心界面，并完成与本地 API 的联调和桌面端运行骨架，为第六阶段测试与发布提供可操作的完整产品形态。

---

## 1. 阶段目标与交付物

### 1.1 核心目标

| 目标 | 说明 |
|------|------|
| **首页完成** | 展示“今日热点”“为你推荐”“最近更新”等核心内容区块，形成产品主入口 |

### 2.3 当前实现约束

当前仓库虽然存在 `index.html` 静态页面，但尚未具备以下正式前端能力：

- 未引入 React / React DOM / React Router
- 未引入前端构建链（如 Vite）
- 未引入桌面容器依赖（Tauri/Electron）
- API 当前仅由 Express 提供，尚未承担前端静态资源托管职责

因此 Phase 5 的第一优先级不是“改现有静态页”，而是建立正式前端工程骨架。

### 2.4 阶段边界

Phase 5 **明确不实现**：

- 多用户账号体系
- 云同步
- 评论、社交分享、消息通知中心
- 推荐算法的进一步升级
- 外部在线部署
- 完整发布流程与大规模性能压测（属于 Phase 6）

## 3. 技术方案设计

### 3.1 前端技术路线

结合总方案与当前仓库状态，第五阶段采用以下路线：

| 层级 | 选型 | 说明 |
|------|------|------|
| 前端框架 | React 18 + TypeScript | 与总方案保持一致，适合组件化页面构建 |
| 构建工具 | Vite | 启动快、配置轻，适合当前从零补建前端工程 |
| 路由 | React Router | 满足首页、搜索、详情、标签、用户中心的多页路由需求 |
| 数据请求 | TanStack Query | 管理 API 请求、缓存、重试和失效 |
| 局部状态 | Zustand 或 React Context | 管理筛选条件、面板开合、偏好草稿等前端局部状态 |
| UI 样式 | Tailwind CSS + Headless UI / Radix UI | 与总方案一致，便于快速构建桌面感界面 |
| 测试 | Vitest + Testing Library | 复用现有测试体系，降低新增工具链成本 |

**选择理由**：

- 当前项目已是 TypeScript 单仓，补入 React + Vite 最平滑
- Vite 比从零引入 Next.js 或 Electron Renderer 直配更轻
- React Query 能减少首页、搜索页、用户中心的重复请求管理代码
- 组件库采用 Headless 方案，避免 UI 被现成样式体系强绑定

### 3.2 桌面端集成路线

总方案推荐 Tauri，但当前仓库还没有桌面容器基础，因此第五阶段采用“两步走”：

1. **先完成 Web UI + 本地 API 的联调闭环**
2. **再补桌面容器 MVP 骨架**

本阶段建议目标：

- 优先确定桌面容器为 **Tauri**
- 完成 Tauri 工程初始化或等价 Electron 骨架初始化
- 能实现“启动桌面壳 -> 启动前端界面 -> 调用本地 API”的开发态闭环
- 生成基础打包脚本，但正式发布和自动更新放到 Phase 6

**为什么这样拆分**：

- 当前项目尚无 React 工程，直接把“前端搭建 + 桌面壳 + 打包发布”一次做完风险过高
- 先验证页面和 API 交互，再封入桌面壳，问题更容易定位

### 3.3 页面信息架构

第五阶段目标页面如下：

| 页面 | 路由建议 | 核心职责 |
|------|---------|---------|
| 首页 | `/` | 展示热点推荐、个性化推荐、最近更新、全局导航 |
| 搜索页 | `/search` | 输入查询、展示结果、查看调研报告 |
| 文章详情页 | `/documents/:id` | 阅读摘要、查看标签、跳转原文、收藏/标记已读 |
| 标签页 | `/tags`、`/tags/:tag` | 浏览标签和标签下内容集合 |
| 用户中心 | `/me` | 管理兴趣、记忆、收藏、历史、偏好 |

### 3.4 首页设计方案

首页建议拆为 5 个模块：

1. **顶部导航区**
   - 全局搜索入口
   - 页面导航：首页 / 搜索 / 标签 / 用户中心
   - 数据刷新入口（可选）

2. **今日热点区**
   - 展示 Top 5 热点内容
   - 每张卡片展示标题、来源、发布时间、摘要片段、标签
   - 预留“查看全部热点”入口

3. **为你推荐区**
   - 展示个性化推荐 10 条左右
   - 强调推荐原因，例如“匹配兴趣：Agents / CLI”
   - 无记忆数据时可自动降级为热点或最近更新

4. **最近更新区**
   - 展示最新内容时间线
   - 支持快速进入详情页

5. **侧边状态区或信息条**
   - 显示今日新增数量
   - 显示推荐缓存状态或最近刷新时间

### 3.5 搜索页设计方案

搜索页建议分为三栏或两栏布局：

- **搜索控制区**
  - 搜索输入框
  - 搜索模式选择：关键词 / 语义 / 混合
  - 来源筛选
  - 标签筛选
  - 时间范围筛选

- **结果列表区**
  - 标题、摘要、来源、标签、发布时间
  - 支持分页或“加载更多”

- **调研报告区**
  - 展示搜索结果生成的报告摘要
  - 当结果较多时，优先展示自动生成的研究概览

### 3.6 文章详情页设计方案

详情页至少展示以下信息：

- 标题
- 作者
- 来源与发布时间
- 领域标签 / 模型标签 / 类型标签
- 中文摘要 / 英文摘要
- 原文链接
- 收藏按钮
- 标记已读按钮
- 相关推荐区（可复用首页推荐卡片，但不是本阶段硬性要求）

详情页的数据来源以现有 `GET /api/documents/:id` 为主；若摘要缺失，可调用现有摘要接口触发补生成。

### 3.7 标签页设计方案

标签页分为两层：

1. **标签总览**
   - 标签云或标签列表
   - 支持按分类查看：领域 / 模型 / 来源 / 类型 / 年份

2. **标签详情**
   - 展示该标签对应的文档列表
   - 支持时间排序与分页

### 3.8 用户中心设计方案

用户中心建议包括 5 个面板：

| 面板 | 数据来源 | 功能 |
|------|---------|------|
| 兴趣标签 | `GET/POST /api/user/preferences` | 查看、增加、删除兴趣标签 |
| 用户记忆 | `GET /api/user/memory` | 查看当前记忆词、支持手动刷新 |
| 浏览历史 | `GET /api/user/history` | 查看已读内容 |
| 收藏夹 | `GET /api/user/favorites` | 查看和移除收藏 |
| 偏好设置 | `GET/POST /api/user/preferences` | 默认语言、摘要长度、每日推荐数 |

第五阶段不要求复杂表单系统，但要求以下体验完整：

- 偏好可编辑并保存
- 记忆可手动刷新
- 收藏和历史可跳转回原文档详情

### 3.9 前端状态与数据流设计

推荐的数据流建议如下：

```text
页面挂载
  -> React Query 请求 API
  -> Query Cache 管理数据复用
  -> 页面组件消费标准化后的 ViewModel
  -> 用户操作（收藏 / 已读 / 偏好修改）
  -> 提交 mutation
  -> 失效相关 query
  -> 局部刷新页面数据
```

建议 query key 设计：

- `["home", "hot"]`
- `["home", "personalized"]`
- `["documents", params]`
- `["document", id]`
- `["search", params]`
- `["tags", category]`
- `["tag-documents", tag, params]`
- `["user", "preferences"]`
- `["user", "memory"]`
- `["user", "history", params]`
- `["user", "favorites", params]`

### 3.10 API 联调策略

Phase 4 已具备大部分前端所需接口，但第五阶段仍需注意以下集成细节：

1. **前端统一 API Client**
   - 封装成功/失败返回结构
   - 统一处理 `success: false`
   - 统一处理分页元信息

2. **必要时补充轻量接口修正**
   - 若页面实现中发现单接口返回结构不足，允许在 Phase 5 中补充少量后端接口优化
   - 但不重新设计推荐和搜索内核

3. **开发态代理**
   - 前端开发服务器通过 proxy 指向本地 Express
   - 避免浏览器跨域问题

4. **桌面态同源调用**
   - 桌面壳中优先走本地 `http://127.0.0.1:<port>` 或内嵌同源访问

## 6. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 前端从零搭建工作量偏大 | 影响 2 周节奏 | 先完成页面骨架和核心功能，复杂视觉优化放后 |
| 桌面容器引入过晚 | 可能拖到联调后期 | Week 2 Day 9 前必须完成最小骨架 |
| API 返回结构与页面需求不完全对齐 | 增加返工 | 允许 Phase 5 做小幅 API 修正，不重做核心后端 |
| 搜索与推荐页面请求较多 | 首屏体验变慢 | 使用 React Query、骨架屏和并行请求 |
| 当前仓库无前端依赖 | 工程初始化跨度大 | 先建立独立前端层，再逐步接入桌面壳 |
