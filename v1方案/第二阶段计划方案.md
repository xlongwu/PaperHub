# PaperHub 第二阶段计划方案

## 目标

在第一阶段基础设施之上，完成全部三个数据源的采集覆盖、LLM 智能摘要生成、自动标签分类，以及用户偏好管理。本阶段结束时，系统应能自动从 arXiv + GPT Blog + Claude Blog 采集内容，为每篇内容生成中文摘要并打上领域标签，同时支持用户配置兴趣偏好。

## 时间

2 周（10 个工作日）

## 任务分解

### Week 1: 多源采集与标签体系

#### Day 1-2: GPT Blog 采集器

**任务 1.1: RSS 解析器**
- 创建 `src/utils/rss.ts`：
  - 使用原生 `fetch` 获取 RSS Feed XML
  - 解析 XML 提取 `<item>` 元素：title, link, pubDate, description/content
  - 处理 RSS 2.0 和 Atom 格式差异
  - 返回标准化的 `RawDocument` 列表

**任务 1.2: GPT Blog 采集器实现**
- 创建 `src/collectors/gpt-blog.ts`：
  - 实现 `ContentCollector` 接口
  - 解析 `https://openai.com/blog/rss.xml`（或实际 RSS URL）

  - 若 RSS 内容不全，再请求原文页面补充（使用 `fetch` 获取 HTML，提取正文）
  - 语言标记为 `en`
  - 去重：采集前检查 `documentExists(url)`

**交付物**: 可独立运行 `tsx src/collectors/gpt-blog.ts` 并采集博客文章到数据库

#### Day 3-4: Claude Blog 采集器

**任务 1.3: 复用并扩展 sitemap 采集**
- 创建 `src/collectors/claude-blog.ts`：

  - 解析 `https://www.anthropic.com/sitemap.xml`（或博客列表页）

  - 使用 `fetch` 获取原文页面，提取正文内容（参考 `web.ts` 的 HTML 解析方式）
  - 语言标记为 `en`
  - 去重：采集前检查 `documentExists(url)`

**交付物**: 可独立运行 `tsx src/collectors/claude-blog.ts` 并采集博客文章到数据库

### Week 2: 摘要流水线与用户系统

#### Day 6-7: LLM 摘要生成

**任务 2.1: 摘要 Prompt 构建**
- 创建 `src/summarizer/prompt.ts`：
  - 复用 `src/prompts.ts` 的 Prompt 构建模式
  - 定义中英文摘要模板：
    ```
    请对以下 {{type}} 进行摘要总结：

    标题：{{title}}
    作者：{{authors}}
    来源：{{source}}
    发布时间：{{publishedAt}}

    原文内容：
    {{content}}

    要求：
    1. 用 {{language}} 输出，字数控制在 200-400 字
    2. 包含：研究背景、核心方法、主要结论、创新点
    3. 文末必须附带引用信息：**来源**: [{{title}}]({{url}})
    ```

**任务 2.2: 摘要流水线实现**
- 创建 `src/summarizer/index.ts`：
  - `summarizeDocument(doc: Document, lang: Lang): Promise<string>`
  - 流程：
    1. 检查 `isSummarized` 标志，已摘要则跳过
    2. 构建 Prompt（使用 `doc.abstract` 或 `doc.title` 作为内容）
    3. 调用 `callLlm(prompt, 2048)` 生成摘要
    4. 存储结果到 `summaryZh` 或 `summaryEn`
    5. 更新 `isSummarized = true`
  - 批量摘要时控制并发（复用 `LLM_CONCURRENCY` 限制器）
  - 成本优化：使用性价比模型（DeepSeek/GPT-4o-mini）

**任务 2.3: 摘要调度集成**
- 修改 `src/scheduler.ts`：
  - 采集完成后自动触发未摘要文档的批量摘要
  - 新增 `runSummarization()` 任务，每 30 分钟检查一次待摘要文档
  - 日志记录摘要进度和成本

**交付物**: 运行 `pnpm start` 后，新采集的论文自动获得中文摘要

## 技术决策记录

### 决策 1: 博客正文获取策略
- **方案 A**: 仅使用 RSS 中的 description（内容少，简单）
- **方案 B**: RSS 获取元数据 + 原文页面 fetch 获取完整内容（内容多，准确）
- **选择**: 方案 B（优先），若原文页面获取失败则降级为方案 A
- **原因**: 摘要质量依赖内容完整性，完整正文有助于 LLM 生成准确摘要

### 决策 2: 标签分类触发时机
- **方案 A**: 采集时立即分类（同步，延迟采集完成）
- **方案 B**: 采集后异步批量分类（独立任务，不阻塞采集）
- **选择**: 方案 B
- **原因**: LLM 调用有延迟和成本，异步处理避免阻塞采集流程；摘要和标签可并行执行

### 决策 3: 摘要内容来源
- **方案 A**: 使用 `abstract` 字段（arXiv 有摘要，博客可能无）
- **方案 B**: 使用 `fullText` 或 `content`（内容多，token 消耗大）
- **选择**: 方案 A 为主，方案 B 为补充
- **原因**: arXiv 摘要质量高，足够生成摘要；博客若无摘要则使用 description 或前 2000 字正文

### 决策 4: LLM 模型选择（摘要 vs 标签）
- **摘要**: 使用性价比模型（DeepSeek-V3 / GPT-4o-mini），2048 tokens
- **标签分类**: 使用轻量模型（GPT-4o-mini / DeepSeek-V3），1024 tokens
- **原因**: 摘要和标签都是高频调用，需控制成本；质量要求可通过 Prompt 优化弥补

## 风险与应对

| 风险 | 可能性 | 影响 | 应对 |
|------|--------|------|------|
| **GPT Blog RSS 地址变更** | 中 | 采集失败 | 实现降级：若 RSS 失败则尝试直接解析博客列表页 HTML |
| **Claude Blog 页面结构改版** | 中 | 内容提取失败 | 使用多种 CSS 选择器尝试提取；记录失败日志便于人工修复 |
| **LLM 摘要成本过高** | 中 | 运行费用高 | 摘要缓存 + 批量控制并发 + 使用性价比模型 |
| **LLM 标签分类不稳定** | 中 | 标签质量差 | 规则映射优先（arXiv），LLM 仅用于博客；提供标签修正接口 |
| **博客内容过长导致 token 超限** | 低 | 摘要生成失败 | 截断内容至 4000 tokens；优先使用 abstract/description |
