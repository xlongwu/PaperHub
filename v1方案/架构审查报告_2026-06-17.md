## 三、Bug 与代码缺陷

### 🔴 P0 — 严重问题

#### 3.1 摘要重复生成风险

**位置**: `src/summarizer/index.ts` 第 15-18 行

```typescript
if (doc.isSummarized) {
  const existing = lang === "zh" ? doc.summaryZh : doc.summaryEn;
  if (existing) return existing;
}
```

**问题**: 若文章已有中文摘要但无英文摘要，`isSummarized=true` 且 `summaryZh` 有值时英文摘要正确跳过。但存在一个场景：摘要生成时 LLM 调用成功、但 `updateDocument` 调用写入 DB 失败（如磁盘满），`isSummarized` 未被持久化，下次仍会重复调用 LLM。虽然浪费有限，但建议在 `summarizeBatch` 中增加 DB 写入成功后的确认。

#### 3.2 `refreshTagStats` 无事务保护 — 崩溃导致数据丢失

**位置**: `src/db/tags.ts` 第 103 行

```typescript
db.prepare("DELETE FROM tag_stats").run();  // 先全部删除
// ... 后续逐条 INSERT ...
```

**问题**: 如果 `DELETE` 执行后、`INSERT` 完成前进程崩溃，`tag_stats` 表将处于空状态，前端标签云显示为空。**应包裹在事务中**。

#### 3.3 `fromDb`/`rowToDocument` 无异常处理 — 崩溃风险

**位置**: `src/db/search.ts` 第 222-237 行, `src/db/documents.ts` 第 45-67 行, `src/db/tags.ts` 第 199-221 行

```typescript
authors: JSON.parse(String(row.authors ?? "[]")),  // 损坏的 JSON 直接抛异常
domainTags: JSON.parse(String(row.domain_tags ?? "[]")),
```

**问题**: 三个 `fromDb` 变体中 `JSON.parse` 均无 try-catch。如果任何 JSON 字段因迁移错误或手动修改而损坏，**进程将崩溃**，且无任何日志指明哪条记录出了问题。

---

### 🟡 P1 — 高优先级

#### 3.4 FTS5 查询注入 — 特殊字符未转义

**位置**: `src/db/search.ts` 第 52-60 行

```typescript
const ftsQuery = terms.join(" AND ");
db.prepare("... WHERE document_fts MATCH ?").run(ftsQuery);
```

**问题**: FTS5 保留字（`AND`, `OR`, `NOT`, 括号等）出现在用户输入中时会改变查询语义。虽然 `better-sqlite3` 使用了参数化查询避免 SQL 注入，但 **FTS5 本身也是一种查询语言**。例如搜索 `agent AND NOT rag` 会执行布尔逻辑，用户未预期。更严重的是，特殊字符可能导致 FTS5 语法错误而搜索失败。

**建议**: 对 FTS5 保留字符进行转义（如用双引号包裹每个 term）。

#### 3.5 `searchVector` 全量 KNN — 性能悬崖

**位置**: `src/db/search.ts` 第 175-181 行

```typescript
const totalIndexed = countIndexedVectors();
const rows = db.prepare(sql).all(serializedEmbedding, totalIndexed);
```

**问题**: `k = totalIndexed` 意味着每次向量搜索都返回全部索引向量，然后在 JavaScript 中过滤。当文档数超过 1 万篇时，每次搜索都加载全部向量到内存并逐条过滤，延迟将从毫秒级飙升到秒级。

**建议**: 设置合理 K 值（如 200），减少数据传输量。

#### 3.6 LLM Provider 创建失败静默 — 缺 API Key 时不报错

**位置**: `src/report.ts` 第 20 行

```typescript
const provider: LlmProvider = createProvider();
```

**问题**: `createProvider()` 在模块加载时执行。如果环境变量未设置，默认使用 `anthropic`，但若 `ANTHROPIC_API_KEY` 也为空，`new AnthropicProvider()` 可以成功创建对象，但首次 `call()` 时才会失败。**错误被延迟到实际调用点**，而不是启动时就快速失败。

#### 3.7 搜索过滤器逻辑重复

**位置**: `src/search.ts` 的 `matchesHybridFilters` vs `src/db/search.ts` 的 `matchesSearchFilters`

两个函数实现几乎相同的过滤逻辑（sources, tags, dateRange），但分别在两个文件中。任何一方的修改必须同步到另一方，否则混合搜索的过滤不一致。

---

### 🔵 P2 — 中优先级

#### 3.8 `stopScheduler` 不等待运行中任务

**位置**: `src/scheduler.ts` 第 211-217 行 + `src/index.ts` 第 55 行

```typescript
stopScheduler();
server.close(() => { ... });
```

**问题**: 优雅关闭时，`stopScheduler()` 只停止 cron 触发，不等待正在执行的采集/摘要任务完成。10 秒超时后强制退出。如果采集器正在写数据库，可能导致部分数据不一致。

#### 3.9 API 无输入验证

**位置**: `src/api/server.ts` 全局

```typescript
limit: parseInt(req.query.limit as string) || 20,  // limit=0 被覆盖为 20
source: req.query.source as string | undefined,       // 无来源值验证
```

**问题**: 所有 API 参数均为 `as string` 强制转换，无可校验（无 Zod/joi）。`limit=0` 会被静默覆盖为 20（ES 中 `parseInt("0") || 20` = `20`）。

#### 3.10 向量搜索降级不通知用户

**位置**: `src/search.ts` 第 63-66 行

```typescript
} catch (e) {
  console.error("[search] Vector search failed:", e);
  // Continue with FTS only
}
```

**问题**: 用户明确选择 `mode: "semantic"` 时，若向量搜索失败，返回的只有空 `vecResults`，用户体验是搜不出任何结果，但没有错误提示。

#### 3.11 摘要 batch 中单篇失败导致并发浪费

**位置**: `src/summarizer/index.ts` 第 60-71 行

```typescript
async function worker(): Promise<void> {
  while (queue.length > 0) {
    const doc = queue.shift()!;
    try {
      await summarizeDocument(doc, lang);
    } catch (err) {
      console.error(`[summarizer] Failed for ${doc.id}: ${err}`);
    }
  }
}
```

**问题**: worker 使用 `while(queue.length > 0)` 循环，如果 `summarizeDocument` 因网络错误卡住（而非抛出异常），worker 将无限期阻塞，不会释放并发槽位。应考虑超时机制。

## 四、安全漏洞

### 🔴 严重

#### 4.1 HTTP API 完全无限速

**位置**: `src/api/server.ts`

- 未安装 `express-rate-limit` 或等效中间件
- `POST /api/collect` 可被无限调用，触发外部 API 调用
- `POST /api/summarize` 可被无限调用，产生 LLM API 费用
- `GET /api/search` 可被无限调用，触发 embedding 生成 + LLM 报告

**风险**: 攻击者在同一局域网中可无限触发 LLM 调用，造成 API 费用损失。

#### 4.2 CORS 完全未配置

**位置**: `src/api/server.ts`

- 未安装 `cors` 中间件
- 任意来源的网页可向 `localhost:3000` 发起跨域请求
- 在共享网络环境中，恶意网站可读取本地 API 数据

**风险**: 数据泄露（共享网络）或 SSRF 类攻击。

### 🟡 高危

#### 4.3 无 `unhandledRejection` / `uncaughtException` 处理器

**位置**: `src/index.ts`

```typescript
main().catch((err) => { ... });
```

**问题**: 只有 `main()` 的拒绝被捕获。Express 5 不会自动捕获路由处理器返回的未处理 Promise。如果内部代码触发未处理的拒绝，进程将崩溃且无日志。

#### 4.4 请求体无大小限制

**位置**: `src/api/server.ts` 第 62 行: `app.use(express.json());`

**问题**: 未设置 `limit` 选项。恶意的大 JSON 请求体可能耗尽内存。

### 🔵 中危

#### 4.5 API 密钥可能在日志中泄露

**位置**: `src/embedding.ts` 第 77 行

```typescript
Authorization: `Bearer ${apiKey}`
```

虽然在正常使用中不会直接打印，但若 HTTP 请求被日志中间件捕获，API 密钥将明文暴露。

#### 4.6 用户偏好键无白名单

**位置**: `src/api/server.ts` 第 396-408 行

```typescript
for (const [key, value] of Object.entries(req.body as Record<string, string>)) {
  setUserPreference(key, value);
}
```

**问题**: 攻击者可写入任意偏好键，包括内部使用的键（如数据库路径、日志路径等），造成配置混乱。

## 五、架构质量评估

### ✅ 优秀实践

1. **模块边界清晰**: `db/` `collectors/` `recommendation/` `api/` `tagger/` `summarizer/` `ui/` 职责分明，依赖方向合理
2. **LLM Provider 抽象层**: 工厂模式 + `ProviderName` 类型安全，新增提供商只需注册
3. **数据库单例 + WAL**: 连接管理安全，读/写无锁冲突
4. **去重机制完善**: URL 精确匹配 + Levenshtein 编辑距离标题去重
5. **LLM 并发控制**: 信号量机制防止 429，429 自动退避重试
6. **Embedding 缓存**: SHA256 文件缓存避免重复 API 调用
7. **前端 API Client**: 统一 fetch 封装，与后端返回结构对齐
8. **渐变关闭**: SIGINT/SIGTERM 处理，10s 超时兜底

### ⚠️ 需改进

1. **三个 `fromDb` 函数重复**: `documents.ts`, `search.ts`, `tags.ts` 各有自己的 `rowToDocument` 实现，应提取共享
2. **日志分散**: `app.log` / `scheduler.log` / `errors/YYYY-MM-DD.log` 三个独立目标，应统一
3. **迁移无事务**: `runMigrations()` 中 `db.exec()` 和 `postMigrate` 回调不在同一事务中
4. **Embedding 绕过了 Provider 体系**: `embedding.ts` 直接 `fetch` OpenAI API，而非复用 `providers/` 的抽象层
5. **`fullText` vs `fullTextPath` 语义不一致**: AGENTS.md 说存储内容字符串，实际是文件路径
6. **混合搜索权重固定**: `FTS 0.6 / Vector 0.4` 不根据结果数量动态调整
