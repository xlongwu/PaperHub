### 2.2 全文搜索模块（FTS5）

**计划要求**（§3.2）：
- 前缀匹配（`term*`）
- BM25 排名
- 来源过滤 + 标签过滤
- 高亮片段

**实际实现**：
- `searchFts5()` 将用户输入拆词并构造成 `term* AND term*`
- 通过 FTS5 `rank` 排序并做分数归一化
- 支持 `sources`、`tags`、`dateRange` 过滤
- 返回高亮摘要片段

**审查结论**: ✅ **满足计划要求**

**说明**：
- 当前高亮片段由应用层 `buildSnippet()` 生成，而非直接调用 FTS5 `snippet()`；功能达成，但实现方式与方案文档示例不同
- 标签过滤已使用 JSON 精确匹配逻辑，避免了早期 `LIKE` 方案可能带来的子串误匹配

---

### 2.3 向量搜索模块（sqlite-vec）

**计划要求**（§3.3）：
- 生成查询 embedding
- sqlite-vec KNN 搜索
- 相似度阈值过滤

**复核前发现的问题**：
- 向量写入使用 JSON 字符串，和当前 sqlite-vec 接口预期不匹配
- 表现为测试失败：插入 embedding 时出现 JSON parsing error

**修复结果**：
- 向量存储改为 `Float32Array` 序列化后的 `Buffer`
- 增加维度校验和有限值清洗
- 增加 `countIndexedVectors()` 便于索引校验和测试
- 向量搜索补上 `sources` / `tags` / `dateRange` 过滤，避免 semantic/hybrid 模式绕过条件

**审查结论**: ✅ **问题已修复，语义搜索可用**

---

### 2.4 搜索编排层（混合搜索）

**计划要求**（§3.4）：
- 关键词 + 语义混合搜索
- 去重、分页、排序
- 可选 LLM 综述报告

**复核前发现的问题**：
- `hybridSearch()` 只将 `sources` / `tags` 传入 FTS 分支，向量分支可能泄漏不符合过滤条件的结果
- 候选集只取 `limit * 2`，然后再做 `slice(offset, offset + limit)`，后续页可能丢结果

**修复结果**：
- `sources` / `tags` / `dateRange` 统一传递到向量检索与最终合并后的防御性过滤
- 使用 `candidateLimit = max(offset + limit, limit * 5, 100)` 的候选窗口策略，显著降低后续页截断风险
- 补充了“标签过滤下 hybrid 结果不泄漏”和“后续页可访问”的回归测试

**审查结论**: ✅ **核心行为符合计划，分页与过滤一致性问题已修复**

**仍保留的简化项**：
- 未实现动态混合权重调整；当前仍使用固定权重 `FTS 0.6 / Vector 0.4`
- 当前实现仍属于本地内存聚合分页，不是严格意义上的全局稳定分页算法；对 Phase 3 数据规模已足够

---

### 2.5 标签系统

**计划要求**（§3.5）：
- 标签统计
- 标签云
- 按标签筛选文档
- 增量维护

**复核前发现的问题**：
- 早期报告中记录了 `source_tag` 匹配异常
- 更深层的结构性问题是 `tag_stats` 使用 `tag` 单列主键，会导致不同类别的同名标签互相覆盖

**修复结果**：
- `getDocumentsByTag()` / `countDocumentsByTag()` 按 domain/model/source/source_tag/type/year 统一匹配
- `tag_stats` 迁移为 `(tag, category)` 复合主键
- `updateTagStatsForDocument()` 的 UPSERT 冲突目标同步调整为 `(tag, category)`
- 新增测试验证“同一 tag 文本在多个 category 中可并存”

**审查结论**: ✅ **标签统计与筛选满足计划要求**
