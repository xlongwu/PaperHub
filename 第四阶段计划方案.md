# PaperHub 第四阶段计划方案：推荐系统与用户记忆

> **阶段定位**：承接第三阶段（搜索与标签系统），聚焦**热点推荐**、**用户记忆提取**、**个性化推荐**三大模块，为第五阶段首页/用户中心界面提供稳定的数据接口与可解释推荐结果。

---

## 1. 阶段目标与交付物

### 1.1 核心目标

| 目标 | 说明 |
|------|------|
| **热点推荐** | 基于发布时间、时间衰减和来源多样性，生成无需用户配置的“今日热点”列表 |
| **用户记忆系统** | 从 `digests/` 历史日报/周报/月报中提取稳定兴趣关键词，形成可持久化的用户记忆条目 |
| **个性化推荐** | 综合兴趣标签、用户记忆、浏览历史、收藏偏好，对新文档进行标签匹配排序并排除已读内容 |
| **推荐 API** | 提供统一接口，向前端返回热点推荐、个性化推荐、用户记忆和推荐解释 |
| **推荐调度与缓存** | 支持定时构建热点榜单、按需/定时刷新用户记忆，避免首页每次加载都全量重算 |

### 1.2 交付物清单

- [ ] `src/recommendation/hot.ts` — 热点推荐打分与多样性重排
- [ ] `src/recommendation/memory.ts` — 从 `digests/` 提取用户记忆关键词
- [ ] `src/recommendation/profile.ts` — 用户画像构建（兴趣标签 + 记忆 + 历史 + 收藏）
- [ ] `src/recommendation/index.ts` — 个性化推荐编排层（候选召回、打分、去重、解释）
- [ ] `src/api/server.ts` — 新增推荐相关 REST API
- [ ] `src/scheduler.ts` — 新增热点榜单构建与用户记忆刷新任务
- [ ] `src/db/index.ts` — 数据库迁移 v4：用户记忆表、推荐快照表
- [ ] 测试覆盖：热点推荐测试、记忆提取测试、个性化推荐测试、推荐 API 集成测试

## 3. 技术方案设计

### 3.1 数据库 Schema 扩展（Migration v4）

```sql
-- 用户记忆条目
CREATE TABLE IF NOT EXISTS user_memory_terms (
  term TEXT NOT NULL,
  category TEXT NOT NULL,          -- 'domain' | 'model' | 'source' | 'topic'
  weight REAL NOT NULL DEFAULT 0,
  source TEXT NOT NULL,            -- 'digest' | 'preference' | 'history' | 'favorite'
  last_seen_at TEXT DEFAULT CURRENT_TIMESTAMP,
  meta TEXT,                       -- JSON: 统计来源、命中文档、原始片段
  PRIMARY KEY (term, category, source)
);

CREATE INDEX IF NOT EXISTS idx_memory_weight ON user_memory_terms(weight DESC);
CREATE INDEX IF NOT EXISTS idx_memory_source ON user_memory_terms(source);

-- 推荐快照
CREATE TABLE IF NOT EXISTS recommendation_snapshots (
  kind TEXT NOT NULL,              -- 'hot' | 'personalized'
  snapshot_date TEXT NOT NULL,     -- YYYY-MM-DD
  document_id TEXT NOT NULL,
  score REAL NOT NULL,
  rank INTEGER NOT NULL,
  reason TEXT,                     -- JSON: 推荐解释
  created_at TEXT DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (kind, snapshot_date, document_id)
);

CREATE INDEX IF NOT EXISTS idx_rec_snapshots_kind_date
  ON recommendation_snapshots(kind, snapshot_date, rank);
```

**设计说明**：
- `user_memory_terms` 用于保存从 `digests/`、用户偏好、历史和收藏提取出的统一兴趣词
- `recommendation_snapshots` 用于缓存热点榜单和最近一次个性化推荐结果，支持首页快速加载与调试复现
- 当前阶段只服务单用户桌面端，不额外引入 `user_id`

### 3.2 热点推荐模块

```typescript
// src/recommendation/hot.ts

interface HotRecommendationOptions {
  limit?: number;                  // 默认 10
  windowDays?: number;             // 默认 14 天
  includeRead?: boolean;           // 默认 true
}

interface RecommendationEntry {
  document: Document;
  score: number;
  reason: {
    kind: "hot" | "personalized";
    matchedTags?: string[];
    factors: string[];
  };
}

export function buildHotRecommendations(options: HotRecommendationOptions): RecommendationEntry[] {
  // 1. 读取最近 N 天文档
  // 2. 计算时间衰减 + 新鲜度加权
  // 3. 对单一来源过度集中的结果做多样性重排
  // 4. 输出可解释原因
}
```

**推荐公式（MVP）**：

```text
hot_score =
  time_decay(publishedAt, half_life = 48h) * 0.75
  + freshness_boost(<24h / <72h) * 0.15
  + summary_bonus(isSummarized) * 0.10
```

**来源多样性控制**：
- 第一次排序按 `hot_score` 降序
- 第二次重排对同一来源设置软上限
- 默认策略：前 10 条中单一来源不超过 5 条；若候选不足则放宽约束

**选择理由**：
- 时间衰减符合总方案“今日热点”的定义
- 多样性重排能避免 arXiv 或单一博客刷屏
- 不依赖外部点击数据，符合单用户桌面端现实

### 3.3 用户记忆提取模块

```typescript
// src/recommendation/memory.ts

interface MemoryExtractionOptions {
  maxDays?: number;                // 默认读取最近 30 天 digests
  maxTerms?: number;               // 默认保留 20 个兴趣词
  useLlmFallback?: boolean;        // 默认 false
}

interface MemoryTerm {
  term: string;
  category: "domain" | "model" | "source" | "topic";
  weight: number;
  source: "digest";
}

export function extractMemoryFromDigests(options?: MemoryExtractionOptions): MemoryTerm[] {
  // 1. 读取 digests/ 下近期日报/周报/月报
  // 2. 提取领域标签、模型词、来源词和高频主题词
  // 3. 做归一化和去重
  // 4. 返回可持久化的 MemoryTerm[]
}
```

**默认提取策略**：

1. **规则优先**：
   - 复用已有标签体系词表（Agents, RAG, Multimodal, Safety 等）
   - 复用 `model-extract.ts` 的模型名正则
   - 识别固定来源词（arXiv, GPT Blog, Claude Blog）
2. **频次加权**：
   - 近 7 天命中权重大于 8~30 天
   - 周报/月报命中权重大于单日报中的偶发命中
3. **可选 LLM fallback**：
   - 当规则提取结果过少（如 < 5 项）时，允许调用 LLM 将原始词归一化为少量稳定主题
   - 默认关闭，优先控制成本

**选择理由**：
- 总方案允许 “LLM 或 TF-IDF” 两种方向
- 当前仓库已有标签词表和模型抽取规则，规则优先最稳
- `digests/` 是 Markdown 文本，先做轻量规则抽取更符合本地优先和成本控制要求

### 3.4 用户画像构建

```typescript
// src/recommendation/profile.ts

interface UserProfile {
  weightedTags: Array<{ term: string; weight: number; source: string }>;
  excludedDocumentIds: Set<string>;
}

export function buildUserProfile(): UserProfile {
  // 1. 读取显式兴趣标签 interest_tags
  // 2. 读取 user_memory_terms
  // 3. 从 favorites / history 反推标签偏好
  // 4. 合并权重并产出排除集合
}
```

**权重来源设计**：

| 来源 | 默认权重 | 说明 |
|------|---------|------|
| 显式兴趣标签（`interest_tags`） | 1.00 | 用户主动声明，权重最高 |
| 收藏推导标签 | 0.85 | 明确信号，优先级高 |
| `digests/` 记忆词 | 0.70 | 反映长期关注主题 |
| 浏览历史推导标签 | 0.35 ~ 0.60 | 需要做时间衰减，避免偶发阅读过度放大 |

**排除策略**：
- 默认排除浏览历史中的文档
- 默认排除已收藏文档
- 热点推荐不排除已读；个性化推荐默认排除已读和已收藏

### 3.5 个性化推荐编排

```typescript
// src/recommendation/index.ts

interface PersonalizedRecommendationOptions {
  limit?: number;
  windowDays?: number;             // 默认 30 天
  includeRead?: boolean;           // 默认 false
}

export async function buildPersonalizedRecommendations(
  options?: PersonalizedRecommendationOptions,
): Promise<RecommendationEntry[]> {
  // 1. 构建用户画像
  // 2. 读取近期候选文档
  // 3. 计算标签 / 模型 / 来源匹配分
  // 4. 加入时间衰减
  // 5. 排除已读/已收藏
  // 6. 输出带解释的推荐列表
}
```

**个性化推荐分数**：

```text
personal_score =
  profile_match_score * 0.65
  + recency_score * 0.25
  + source_bonus * 0.05
  + summary_bonus * 0.05
```

**匹配逻辑**：
- `domainTags` 命中显式兴趣标签或记忆词时加权最高
- `modelTags` 命中模型偏好时加分
- `source` / `sourceTag` 命中稳定偏好来源时小幅加分
- 推荐结果必须附带 `matchedTags` 与 `factors`，便于前端展示“推荐原因”

**候选范围**：
- 默认只从最近 30 天文档中选取
- 若候选不足，再回溯至最近 90 天

### 3.6 API 设计

新增推荐相关接口：

| 路由 | 方法 | 说明 |
|------|------|------|
| `/api/recommendations/hot` | GET | 获取热点推荐列表 |
| `/api/recommendations/personalized` | GET | 获取个性化推荐列表 |
| `/api/recommendations/refresh-hot` | POST | 手动重建热点榜单 |
| `/api/user/memory` | GET | 获取当前用户记忆词 |
| `/api/user/memory/rebuild` | POST | 从 `digests/` 手动刷新用户记忆 |

**响应结构建议**：

```json
{

  "data": [
    {
      "document": { "...": "..." },
      "score": 0.92,
      "reason": {
        "kind": "personalized",
        "matchedTags": ["Agents", "RAG"],
        "factors": ["interest_tags:Agents", "memory:RAG", "recently_published"]
      }
    }
  ],
  "meta": {
    "limit": 10,
    "total": 10
  }
}
```

### 3.7 调度与缓存

**推荐调度策略**：

| 任务 | 周期 | 说明 |
|------|------|------|
| 热点榜单生成 | 每日 00:10 | 计算当日热点并写入 `recommendation_snapshots` |
| 用户记忆刷新 | 每日 00:20 | 从近 30 天 `digests/` 重新提取兴趣词 |
| 个性化推荐缓存 | 启动时或手动触发 | 单用户桌面端可按需构建；若前端频繁请求再考虑缓存 |

**缓存策略**：
- 热点榜单必须持久化，首页读取时避免重复计算
- 个性化推荐默认按需生成；若一次会话内频繁调用，可引入进程内短缓存（5~15 分钟）
- 记忆词刷新后应主动失效个性化推荐缓存

## 5. 测试与验收标准

### 5.1 功能验收

| 验收项 | 目标 |
|--------|------|
| 热点推荐 | 能返回最近内容中的 Top N，并体现时间衰减与来源多样性 |
| 用户记忆 | 能从真实 `digests/` 中提取稳定兴趣词并持久化 |
| 个性化推荐 | 能基于兴趣标签、记忆、收藏和历史生成推荐结果 |
| 已读排除 | 个性化推荐默认不返回浏览历史和已收藏文档 |
| 推荐解释 | 每条推荐结果都包含匹配标签和推荐因素 |
| 推荐 API | 首页所需热点/个性化/记忆接口可直接调用 |

### 5.2 测试清单

- `recommendation.test.ts`
  - 时间越近的文档热点分越高
  - 同一来源过多时会被多样性重排
  - 候选不足时不应强行丢弃内容

- `memory.test.ts`
  - 从示例 `digests/` 中正确提取领域标签和模型词
  - 重复词可合并权重
  - 超过上限时按权重截断

- `api.test.ts` / `api-recommendation.test.ts`
  - 热点接口返回非空结构
  - 个性化推荐接口默认排除已读文档
  - 用户记忆刷新接口可重建条目

### 5.3 性能目标

| 操作 | 目标 |
|------|------|
| 热点榜单构建（最近 14 天） | < 500ms |
| 用户记忆提取（最近 30 天 digests） | < 2s |
| 个性化推荐生成（30 天候选） | < 800ms |
| 热点推荐接口响应 | < 100ms（命中快照） |

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| `digests/` 内容风格不稳定 | 记忆提取质量波动 | 规则优先 + LLM fallback，保留人工可解释词表 |
| 新用户历史过少 | 个性化推荐冷启动差 | 使用 `interest_tags` + 热点推荐兜底 |
| 历史与收藏数据过少 | 画像过于稀疏 | 提高显式兴趣标签和 digest 记忆权重 |
| 热点榜单被单一来源主导 | 首页体验差 | 增加来源多样性重排 |
| 推荐构建重复计算过多 | 首页加载慢 | 热点榜单持久化，个性化结果短缓存 |
