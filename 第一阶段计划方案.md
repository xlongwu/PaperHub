# PaperHub 第一阶段计划方案

## 目标

搭建 PaperHub 桌面端应用的完整骨架，实现核心基础设施：本地存储、API 服务、定时调度、以及 arXiv 数据采集与入库。本阶段结束时，应能通过命令行或简单 API 触发 arXiv 采集，并将论文元数据持久化到本地 SQLite 数据库。

## 时间

2 周（10 个工作日）

---

## 任务分解

### Week 1: 项目骨架与基础设施

#### Day 1-2: 项目初始化与依赖搭建

**任务 1.1: 初始化项目结构**
- 创建 `PaperHub/` 目录，初始化 `package.json`
- 配置 TypeScript (`tsconfig.json`)、ESLint、Prettier
- 安装核心依赖：`better-sqlite3`, `node-cron`, `express` (或 `fastify`), `js-yaml`, `dotenv`
- 安装开发依赖：`tsx`, `vitest`, `@types/node`, `typescript`
- 创建基础目录结构：`src/`, `src/collectors/`, `src/db/`, `src/recommender/`, `src/__tests__/`, `digests/`

**交付物**: 可运行 `pnpm install` 和 `pnpm typecheck` 的空项目骨架

**任务 1.2: 从 agents-radar 迁移复用代码**
- 复制以下文件到 `src/`（保持原结构，后续扩展）：
  - `src/providers/` 目录（完整 LLM Provider 层）
  - `src/arxiv.ts`（arXiv 采集器）
  - `src/web.ts`（sitemap 采集逻辑）
  - `src/prompts.ts`（Prompt 构建器）
  - `src/i18n.ts`（国际化框架）
  - `src/date.ts`（日期工具）
  - `src/config.ts`（配置加载器）
  - `src/report.ts`（LLM 调用器）
- 调整导入路径，确保编译通过
- 移除 agents-radar 特有的代码（GitHub API、HN、Trending 等无关模块）

**交付物**: `pnpm typecheck` 通过，复用代码无编译错误

#### Day 3-4: SQLite 数据库层

**任务 1.3: 数据库连接与 Schema**
- 创建 `src/db/index.ts`：
  - 初始化 `better-sqlite3` 连接
  - 数据库文件路径：`~/PaperHub/data/paperhub.db`
  - 创建 `documents` 表（按方案文档 Schema）
  - 创建 `user_preferences` 表
  - 创建 `browsing_history` 表
  - 创建 `favorites` 表
- 实现数据库迁移机制（简单的版本控制）

**交付物**: 数据库可初始化，Schema 正确创建

**任务 1.4: 文档 CRUD 操作**
- 创建 `src/db/documents.ts`：
  - `insertDocument(doc: Document): string` — 插入文档，返回 ID
  - `getDocumentById(id: string): Document | null`
  - `getDocumentsBySource(source: string): Document[]`
  - `getDocumentsByTimeRange(start: string, end: string): Document[]`
  - `updateDocument(id: string, updates: Partial<Document>): void`
  - `documentExists(url: string): boolean` — URL 去重检查
- 所有方法使用同步 API（`better-sqlite3` 的 `.prepare().run()`）

**交付物**: 单元测试通过（`pnpm test`）

#### Day 5: 采集器接口与 arXiv 适配

**任务 1.5: 定义采集器统一接口**
- 创建 `src/collectors/types.ts`：
  ```typescript
  interface RawDocument {
    title: string;
    source: string;
    url: string;
    publishedAt: string;
    authors: string[];
    abstract: string;
    content?: string;
    language: 'zh' | 'en';
    metadata?: Record<string, unknown>;
  }

  interface ContentCollector {
    id: string;
    name: string;
    sourceTag: string;
    interval: number; // milliseconds
    fetch(): Promise<RawDocument[]>;
  }
  ```

**任务 1.6: 适配 arXiv 采集器**
- 创建 `src/collectors/arxiv.ts`：
  - 复用原 `src/arxiv.ts` 的 `fetchArxivData()` 逻辑
  - 包装为 `ContentCollector` 接口
  - 查询分类：cs.AI, cs.CL, cs.LG, cs.CV, cs.IR
  - 将 arXiv 返回数据映射为 `RawDocument`
  - 实现去重：采集前检查 `documentExists(url)`

**交付物**: 可独立运行 `tsx src/collectors/arxiv.ts` 并采集论文到数据库

---

### Week 2: 服务层、调度器与集成测试

#### Day 6-7: API 服务层

**任务 2.1: Express API 服务器**
- 创建 `src/server.ts`：
  - 使用 Express 启动 HTTP 服务（端口默认 3000）
  - 实现路由：
    - `GET /health` — 健康检查
    - `GET /api/documents` — 分页获取文档列表（支持 `source`, `from`, `to` 过滤）
    - `GET /api/documents/:id` — 获取单篇文档详情
    - `POST /api/collect` — 手动触发采集（指定 `source` 或 `all`）
    - `GET /api/stats` — 统计信息（总文档数、各来源数量、今日新增）
  - 统一错误处理和响应格式

**交付物**: `curl http://localhost:3000/api/documents` 返回正确 JSON

**任务 2.2: 数据转换与标准化**
- 创建 `src/collectors/transformer.ts`：
  - `rawToDocument(raw: RawDocument): Document` — 将 `RawDocument` 转换为标准 `Document`
  - 生成 ID：`hash(source + url)`
  - 设置默认值：`typeTag`（根据 source）、`yearTag`（从 publishedAt 提取）、`language`
  - 初始标签：`sourceTag` = source, `domainTags` = []（第二阶段填充）

**交付物**: 转换后的 Document 符合 Schema 定义

#### Day 8: 定时调度器

**任务 2.3: node-cron 调度器**
- 创建 `src/scheduler.ts`：
  - 使用 `node-cron` 注册定时任务
  - arXiv 采集：`0 0 */6 * * *`（每 6 小时）
  - 调度器启动时立即执行一次全量采集（便于开发测试）
  - 提供 `startScheduler()` 和 `stopScheduler()` 方法
  - 采集日志记录到控制台和本地文件 (`~/PaperHub/logs/scheduler.log`)

**任务 2.4: 应用入口整合**
- 修改 `src/index.ts`：
  - 启动时初始化数据库
  - 启动 API 服务器
  - 启动定时调度器
  - 实现优雅关闭（SIGINT/SIGTERM 时关闭数据库和调度器）

**交付物**: 运行 `pnpm start` 同时启动 API 和调度器，arXiv 自动采集并入库

#### Day 9: 去重与错误处理

**任务 2.5: 完善去重逻辑**
- 在 `src/collectors/arxiv.ts` 中实现：
  - URL 精确匹配去重（查询 `documentExists`）
  - 标题相似度去重（编辑距离 < 0.3 时跳过）
- 创建 `src/utils/dedup.ts`：
  - `levenshteinDistance(a: string, b: string): number`
  - `isSimilarTitle(a: string, b: string): boolean`

**任务 2.6: 采集错误处理**
- 实现重试机制：采集失败时重试 3 次，指数退避（1s / 2s / 4s）
- 错误日志记录到 `~/PaperHub/logs/errors/`
- 部分失败不阻断其他采集器（为后续多源采集预留）

**交付物**: 模拟网络故障时，重试机制正常工作

#### Day 10: 集成测试与文档

**任务 2.7: 端到端测试**
- 编写测试脚本验证完整流程：
  1. 清空测试数据库
  2. 触发 arXiv 采集
  3. 验证数据库中有新文档
  4. 调用 API 获取文档列表
  5. 验证响应格式正确
- 测试覆盖：正常采集、重复采集（去重）、网络错误（重试）

**任务 2.8: 第一阶段文档**
- 更新 `CLAUDE.md`：补充第一阶段已实现的能力
- 编写 `docs/phase1.md`：
  - 已完成的模块清单
  - API 接口文档
  - 数据库 Schema 说明
  - 已知限制和待办事项

**交付物**: 所有测试通过，`pnpm test` 绿色

## 技术决策记录

### 决策 1: 桌面端框架暂不选定
- **原因**: 第一阶段聚焦后端基础设施，前端 UI 在第二阶段才涉及
- **行动**: 先用纯 Node.js 进程运行，第二阶段再集成 Electron/Tauri 壳

### 决策 2: 向量检索延后到第三阶段
- **原因**: 第一阶段数据量小，FTS5 全文搜索足够；sqlite-vec 集成在搜索阶段统一实现
- **行动**: 第一阶段仅创建 `documents` 表，暂不创建 `document_embeddings` 虚拟表

### 决策 3: 标签体系延后到第二阶段
- **原因**: 标签分类需要 LLM 调用，第一阶段聚焦数据采集和存储
- **行动**: `domainTags` 初始为空数组，第二阶段填充

### 决策 4: 摘要生成延后到第二阶段
- **原因**: LLM 摘要依赖标签体系（用于 Prompt 上下文），且需要成本优化设计
- **行动**: `summaryZh`/`summaryEn` 初始为空，第二阶段实现 `summarizer.ts`

## 风险与应对

| 风险 | 可能性 | 影响 | 应对 |
|------|--------|------|------|
| arXiv API 限流 | 中 | 采集失败 | 实现指数退避重试；降低初始采集频率 |
| better-sqlite3 原生模块编译失败 | 低 | 数据库不可用 | 预装 Python 和 build tools；提供预编译二进制方案 |
| 复用代码与现有逻辑耦合 | 中 | 迁移困难 | 逐步剥离，先复制再重构；保留原始文件备份 |
| 数据目录权限问题 | 低 | 无法写入数据库 | 检测并提示用户选择数据目录；默认使用用户 home 目录 |
